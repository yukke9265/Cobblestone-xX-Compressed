# クライアントとサーバーの基本

この文書では、Minecraft / NeoForge の mod 開発で最初につまずきやすい
「クライアントとサーバーの役割の違い」を根本から整理します。

GUI、BlockEntity、イベント、同期、右クリック処理などを理解するには、
まずこの土台が分かっていることが大事です。

## 1. 先に結論

Minecraft では、ゲームは大きく次の 2 つに分かれます。

### クライアント

- 画面を描く
- 音を鳴らす
- キーボードやマウス入力を受ける
- プレイヤーに見せるための処理をする

### サーバー

- ワールドの本当の状態を持つ
- ブロックやエンティティの状態を進める
- インベントリや体力などの本当の値を管理する
- 何が起きたかを最終的に決める

つまり、かなり乱暴に言うと次の理解で始めると分かりやすいです。

- クライアント: 見せる側
- サーバー: 決める側

## 2. なぜ分かれているのか

Minecraft は、もともとマルチプレイを前提にした作りです。

もし全員が自分の画面の中だけで勝手に「ブロックを壊した」「アイテムが増えた」「進行バーが終わった」と決めてしまうと、
人によって見えている状態がずれてしまいます。

そこで、**本当のゲーム状態はサーバーが持つ** という形になっています。

クライアントは、その結果を受け取って表示します。

## 3. シングルプレイでもサーバーはいる

ここが最初の大事なポイントです。

シングルプレイでも、内部では次の 2 つが動いています。

1. クライアント
2. 統合サーバー

「1 人で遊んでいるのに server って何？」と感じやすいですが、
シングルプレイは **クライアントとサーバーが同じ PC の中で両方動いている状態** です。

そのため、シングルプレイでたまたま動いたコードでも、
役割を混同していると専用サーバーや同期で問題が出ます。

## 4. 専用サーバーには何がないか

専用サーバーには、次のようなクライアント専用要素がありません。

- 画面
- テクスチャ描画
- `Minecraft.getInstance()`
- Screen
- クライアント専用入力

だから、サーバーでも読み込まれる共通クラスから、
クライアント専用のクラスや API に直接触ると壊れます。

このプロジェクトでいうと、[src/main/java/com/yukke9265/cobblestone_xx_compressed/CobblestonexXCompressedClient.java](src/main/java/com/yukke9265/cobblestone_xx_compressed/CobblestonexXCompressedClient.java) は
`Dist.CLIENT` に限定されているので、クライアント専用コードの置き場所として安全です。

## 5. 何をどちらへ置くか

初心者向けには、まず次の表で考えると整理しやすいです。

### クライアントへ置くもの

- Screen の描画
- パーティクル表示
- モデルや見た目の登録
- クライアント専用の GUI 部品
- メニューに対応する Screen の登録

### サーバーへ置くもの

- BlockEntity の本当の状態
- インベントリの中身
- レシピ進行
- ブロック設置や破壊の最終結果
- GUI を開く決定

### 共通として扱うもの

- ブロック登録
- アイテム登録
- BlockEntity 型の登録
- MenuType 登録
- ワールド内のルールそのもの

## 6. GUI を例にすると分かりやすい

GUI は client / server の違いが一番見えやすいです。

### Screen はクライアント側

Screen は「見た目」です。

- 背景を描く
- 文字を描く
- ボタンを置く
- マウスホバーを表示する

つまり Screen は、プレイヤーへ見せるための層です。

### Menu はサーバー側の橋渡し

Menu は、GUI のための中継役です。

- スロット構成を持つ
- BlockEntity や ContainerData の値を扱う
- プレイヤーインベントリと機械をつなぐ

### BlockEntity は本当の状態を持つ

たとえば炉なら、次の値は BlockEntity 側が本体です。

- 燃焼時間
- 進行度
- 入力スロット内容
- 出力スロット内容

つまり GUI の流れはこうです。

1. サーバー側の BlockEntity が本当の状態を持つ
2. Menu がその状態を GUI 用に橋渡しする
3. クライアント側の Screen が見た目として表示する

## 7. 「右クリックで GUI を開く」とき本当に起きていること

プレイヤーがブロックを右クリックすると、感覚的には「画面が開いた」だけに見えます。

でも実際には、次の流れです。

1. プレイヤー操作が発生する
2. サーバー側でその操作を受ける
3. サーバー側が `openMenu(...)` を呼ぶ
4. 対応する Menu が作られる
5. クライアント側でその MenuType に対応する Screen が表示される

ここで大事なのは、**Screen を直接開いているわけではない** という点です。

サーバーが「この Menu を開く」と決め、
クライアントが「この MenuType ならこの Screen を出す」と従っています。

## 8. このプロジェクトの例

今のプロジェクトでは、役割分担が次のように見えます。

### 共通側の入口

[src/main/java/com/yukke9265/cobblestone_xx_compressed/CobblestonexXCompressed.java](src/main/java/com/yukke9265/cobblestone_xx_compressed/CobblestonexXCompressed.java)

ここでは主に次をしています。

- ブロック登録
- アイテム登録
- BlockEntity 登録
- MenuType 登録

これは基本的に、クライアントだけに閉じた処理ではないので共通側です。

### クライアント専用の入口

[src/main/java/com/yukke9265/cobblestone_xx_compressed/CobblestonexXCompressedClient.java](src/main/java/com/yukke9265/cobblestone_xx_compressed/CobblestonexXCompressedClient.java)

ここでは主に次をしています。

- 設定画面の extension point 登録
- MenuType と Screen の対応付け

これは画面表示に関わるので、クライアント専用側に置くのが正しいです。

## 9. よくある誤解

### 誤解 1: シングルプレイなら全部クライアントでよい

違います。

シングルプレイでも内部ではサーバーが動いています。

### 誤解 2: 画面を開きたいから Screen を直接 new すればよい

機械 GUI では通常そうしません。

まずサーバー側で Menu を開き、クライアント側では対応する Screen が表示されます。

### 誤解 3: 進行度表示がほしいから Screen が値を持てばよい

本当の進行度は BlockEntity やサーバー側ロジックが持つべきです。

Screen はその値を見せるだけです。

### 誤解 4: ボタンを押したら Screen の中でそのまま状態変更すればよい

見た目のクラスだけで本当の状態を変えるのは危険です。

本当に変えたい値がワールド状態なら、最終的にはサーバー側へ渡す必要があります。

## 10. 何が「本当の値」なのかを意識する

client / server を理解するコツは、
その値が **本当のゲーム状態か、表示用の見え方か** を分けて考えることです。

### 本当のゲーム状態

- インベントリ中身
- ブロックの存在
- BlockEntity の内部値
- エンティティの体力
- レシピ進行度

これらはサーバーが基準です。

### 表示用の情報

- 進行バーの長さ
- GUI の背景
- ボタンの見た目
- パーティクル
- ツールチップ

これらはクライアント側の仕事です。

## 11. `level.isClientSide` は何を見ているのか

よく出てくる `level.isClientSide` は、
「今動いている処理がクライアント側の world かどうか」を見るための条件です。

たとえば次のように使います。

```java
if (level.isClientSide) {
    return InteractionResult.SUCCESS;
}
```

これは「クライアント側では見た目上の成功だけ返し、
本当に状態を決める処理はサーバー側でやる」という整理です。

### これを入れる理由

同じ処理が client / server の両方で走る場面では、
どちらで本処理を行うかをはっきり分けないと次のような問題が起きます。

- 二重実行
- 表示だけ変わって本体が変わらない
- 専用サーバーでクラッシュする
- 同期がずれる

## 12. mod 開発での実務上の判断基準

コードを書くとき、迷ったら次の問いで切ると判断しやすいです。

### 問い 1: これは見た目か？

見た目なら、まずクライアント側を疑います。

例:

- GUI 描画
- モデル表示
- エフェクト表示

### 問い 2: これはワールドの本当の状態か？

本当の状態なら、まずサーバー側を疑います。

例:

- ブロックの ON/OFF
- 燃焼時間の減少
- アイテムの消費
- インベントリ内容の変更

### 問い 3: これは client と server をつなぐ橋渡しか？

その場合は Menu や同期用データ、あるいは通信が必要になることがあります。

## 13. つまずいたときの見分け方

不具合が出たときは、症状からある程度切り分けできます。

### 画面が出ない

クライアント側の Screen 登録や MenuType 対応を疑います。

### 画面は出るが値が変わらない

サーバー側の BlockEntity や Menu 側の値更新を疑います。

### シングルでは動くが専用サーバーで壊れる

共通コードからクライアント専用 API を触っている可能性が高いです。

### 一瞬表示が変わるが戻る

クライアント側だけで表示変更していて、
本当の値をサーバー側で変えていない可能性があります。

## 14. 最初の理解として大事な 1 文

初心者向けには、まずこの 1 文を基準にすると整理しやすいです。

**サーバーがゲームの本当の状態を持ち、クライアントはそれを見せる。**

ここから外れそうなコードを書いているときは、
「それは本当に client の仕事か、server の仕事か」を考え直すと戻りやすいです。

## 15. このプロジェクトで今後よく出る判断

この mod で今後よく出る判断を、先に並べると次のようになります。

### 炉の燃焼時間を減らす

サーバー側です。

### 燃焼時間をバーで見せる

クライアント側の Screen です。

### GUI を開くと決める

サーバー側です。

### どの Menu にどの Screen を対応させるか登録する

クライアント側です。

### BlockEntity の保存と読み込み

サーバー側の本体データです。

### パーティクルだけ見せる

クライアント側です。

## 16. まず覚えるとよい順番

client / server を一気に完璧に覚えるのは難しいので、
最初は次の順で押さえるのがおすすめです。

1. Screen は見た目
2. Menu は橋渡し
3. BlockEntity は本体データ
4. サーバーが本当の状態を決める
5. クライアントはそれを表示する

この順で理解すると、GUI だけでなく、ブロック処理や同期も追いやすくなります。
