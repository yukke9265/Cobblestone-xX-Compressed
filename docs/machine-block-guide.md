# 機械系ブロック手順

この文書では、このプロジェクトで機械系ブロックを作るときの全体像をまとめます。

ここでいう機械系ブロックは、たとえば次のようなものです。

- 独自かまど
- 圧縮機
- 粉砕機
- エネルギーを持つ装置
- GUI を開くブロック

今の `COBBLESTONE_FURNACE` は名前としては機械っぽいですが、実装上はまだ普通のブロックです。
本当に機械として動かすには、通常ブロックよりかなり要素が増えます。

## 1. 機械系ブロックで増える主な要素

最低限でも次のような要素が出てきます。

1. 独自 `Block` クラス
2. `BlockEntity` クラス
3. `BlockEntityType` の登録
4. 必要なら `Menu` クラス
5. 必要なら GUI 画面クラス
6. インベントリや処理進行の保存
7. 状態同期

つまり、単純ブロックより 1 段階どころではなく構成が増えます。

## 2. まず知っておくべき役割分担

機械系ブロックでは、役割を分けて考えると理解しやすいです。

### Block

- ワールド上に置かれる見た目と基本ルール
- 右クリック時の入口
- 向きや点灯状態などの BlockState

### BlockEntity

- 中に持つデータ
- 処理の進行状況
- インベントリ内容
- 毎 tick ごとの処理

### Menu / Screen

- GUI を開く仕組み
- サーバーとクライアントの画面連携

最初に大事なのは、**機械の中身は BlockEntity に持たせる**という考え方です。

## 3. いきなり GUI 付きにしない方がよい理由

初心者のうちは、最初から GUI 付き機械を作ると調べる面が一気に増えます。

そのため、この順がおすすめです。

1. 状態付きブロックを作る
2. BlockEntity を持つだけのブロックを作る
3. tick 処理だけ持つ機械を作る
4. 最後に GUI を付ける

## 4. 最小の機械系ブロック構成

最初の機械系ブロックとしては、「時間経過で点灯状態が変わるだけ」のようなものが分かりやすいです。

この段階で必要になるのは次のような構成です。

- `CustomMachineBlock`
- `CustomMachineBlockEntity`
- `ModBlockEntities` のような登録クラス

まだ GUI やレシピ処理までは入れません。

## 5. Block 側でやること

機械ブロックの `Block` クラス側では、たとえば次を担当します。

- 向きプロパティを持つ
- 点灯状態を持つ
- 右クリック時の処理入口を持つ
- `newBlockEntity(...)` で BlockEntity を返す

特に「その座標にひも付く本体データ」を使うなら、BlockEntity とつなぐ構成が必要です。

## 6. BlockEntity 側でやること

`BlockEntity` 側には、たとえば次のようなものを持たせます。

- 進行ゲージ
- 燃焼時間
- 内部アイテムスロット
- エネルギー量
- 処理中かどうか

つまり、機械の「中身」はこちらに置きます。

## 7. 登録面で増えるもの

単純ブロックでは `ModBlocks` と `ModItems` だけでも進めやすいですが、機械系では `BlockEntityType` の登録場所も必要になります。

たとえば今後は次のようなクラス分けが分かりやすいです。

- `ModBlocks`
- `ModItems`
- `ModBlockEntities`

構成を早めに分けておくと、後で GUI まで増やしても追いやすいです。

## 8. 右クリックで何をするか

機械ブロックでは、右クリック時に次のどれかをすることが多いです。

- GUI を開く
- 状態を切り替える
- 内部処理を開始する
- 情報をチャット表示する

最初は GUI を開かず、右クリックで状態が変わる程度から始めると分かりやすいです。

## 9. `lit` 状態を持つと機械らしくなる

機械ブロックでは、動作中かどうかを `BlockStateProperties.LIT` で持たせるのが分かりやすいです。

これを使うと、たとえば次のような表現ができます。

- 動作中だけ明るくなる
- 動作中だけ front テクスチャが変わる
- 停止時と稼働時でモデルを切り替える

## 10. GUI を開く前にやるべきこと

GUI 実装は便利ですが、その前に次の段階を踏んだ方が安全です。

1. 機械ブロックが置ける
2. 向きが合う
3. BlockEntity が作られる
4. 状態を切り替えられる
5. 保存と読み込みができる

この順を通してから GUI に進むと、どこで壊れているか切り分けやすいです。

## 11. 独自かまどを作りたい場合

名前どおりの独自かまどを本当に作るなら、必要な面はさらに増えます。

- 燃料処理
- 入力スロット
- 出力スロット
- 焼成進行
- レシピ参照
- GUI
- 同期

そのため、最初の機械系ブロックとしては少し重めです。

もし最初に作るなら、「見た目はかまどだが、まずは点灯だけするブロック」から始める方が安全です。

## 12. このプロジェクトでのおすすめ順

今の構成から機械系へ進むなら、次の順がおすすめです。

1. 向き付きブロックを作る
2. `lit` 状態を持たせる
3. BlockEntity を追加する
4. 一定時間で状態が変わるだけの機械を作る
5. 必要なら GUI を付ける

この順なら、今ある `ModBlocks` ベースの理解をそのまま伸ばしやすいです。

## 13. よくあるミス

### Block だけで全部やろうとする

機械の内部データまで Block に持たせると整理しづらくなります。中身は BlockEntity に寄せる方がよいです。

### GUI から先に作り始める

土台の BlockEntity や状態管理が固まる前だと、原因切り分けが難しくなります。

### 見た目と内部状態の役割を混同する

見た目の切り替えは BlockState、内部データは BlockEntity、と分けて考えるのが基本です。

## 14. まず作るならこんな機械が向いている

初心者向けには、次のような段階が分かりやすいです。

1. 右クリックで点灯 / 消灯する機械ブロック
2. 数秒だけ点灯し、その後消える機械ブロック
3. アイテム 1 スロットだけ持つ簡単な機械ブロック

本格かまどや多スロット処理機は、そのあとで十分です。
