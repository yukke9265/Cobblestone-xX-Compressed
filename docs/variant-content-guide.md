# 色違い・Tier違い素材の増やし方

この文書では、このプロジェクトで**色違い**や**Tier違い**のアイテム・ブロックを複数作るときに、どこまでを自動化し、どこを手作業にするかを整理します。

たとえば次のようなものを増やしたい場合を想定しています。

- copper tier の剣
- iron tier の剣
- gold tier の剣
- 色だけ違う結晶アイテム
- Tier ごとに色が違う圧縮ブロック

## 1. まず結論

この手の追加は、**全部を同じ方法で自動生成するわけではありません**。

大きく分けると、次の 3 系統に分かれます。

1. Java の登録コード
2. JSON 系リソース
3. PNG テクスチャ

このうち、自動化しやすさは次のように考えると分かりやすいです。

- Java 登録: 自動化しやすい
- JSON: とても自動化しやすい
- PNG テクスチャ: 別スクリプトが必要になりやすい

つまり、**登録と JSON は Mod 側の datagen でまとめ、画像だけ別途扱う**のが一番実用的です。

## 2. 何が datagen 向きか

datagen が向いているのは、規則的に並ぶテキストファイルです。

たとえば次はかなり自動化しやすいです。

- item model
- block model
- blockstates
- lang
- recipe
- loot table
- tags

理由は、名前規則や参照先が一定なら、Java 側でループして生成しやすいからです。

## 3. 何が datagen だけでは作りにくいか

PNG テクスチャそのものは、通常の datagen だけではあまり扱いません。

たとえば次のような画像です。

- `textures/item/red_crystal.png`
- `textures/item/blue_crystal.png`
- `textures/block/copper_machine_block.png`

これらは、Minecraft / NeoForge の通常の data generator の担当範囲というより、**画像編集や画像変換**の範囲です。

そのため、画像は次のどちらかで扱うことが多いです。

1. 手作業で描く
2. 外部スクリプトで色差分を生成する

## 4. いちばんおすすめの分担

色違い・Tier違い素材をたくさん増やすなら、次の分担が一番管理しやすいです。

### Mod 側で管理するもの

- 定義一覧
- Java 登録
- JSON 生成

### 外部で管理するもの

- テクスチャ PNG の色差分生成

この分け方にすると、Minecraft 側の仕様変更と、画像生成処理を分けて考えられます。

## 5. まず最初に決めること

自動化する前に、先に次を決めておくと作業がかなり楽になります。

1. 名前規則
2. Tier 一覧
3. 色の決め方
4. アイテムとブロックの対応ルール

たとえば次のような規則です。

- `copper_crystal`
- `iron_crystal`
- `gold_crystal`
- `diamond_crystal`

ブロックも同じ規則にすると追いやすいです。

- `copper_crystal_block`
- `iron_crystal_block`
- `gold_crystal_block`
- `diamond_crystal_block`

## 6. 定義を 1 か所にまとめる考え方

規則的に増える内容は、まず **定義一覧を 1 か所にまとめる** のが基本です。

やり方としては、次のどちらかが分かりやすいです。

1. `enum` にまとめる
2. 単純な定義クラスや record の一覧にまとめる

たとえば 1 件の定義が次の情報を持つ形です。

- 登録名の接頭辞
- 表示名
- 色
- Tier 数値
- 素材性能

この一覧を使って、登録・モデル・言語・レシピを同時に回せるようになります。

## 7. Java 登録を自動化する考え方

アイテムやブロックそのものの登録は、厳密には datagen ではなく Java 側の登録処理です。

ただし、考え方は同じです。

1. 一覧を作る
2. その一覧を順に登録する

つまり、毎回 `ModItems.java` や `ModBlocks.java` に 1 行ずつ手で足すより、
**「Tier 定義を追加したら登録も増える」構成**にした方が規則素材には向いています。

## 8. JSON 自動生成に向いているもの

次のファイルは、色違い・Tier違いが増えるほど datagen の効果が大きくなります。

### item model

ほとんどの通常アイテムは、ファイル名とテクスチャ参照先だけが違う形になりやすいです。

### blockstates

単純ブロックなら、ほぼ同じ構造でモデル名だけ違うことが多いです。

### block model

`cube_all` 系なら差分はテクスチャ名だけで済みやすいです。

### lang

表示名は `Copper Crystal`、`Iron Crystal` のように規則が作りやすいです。

### recipe

Tier ごとに材料だけ違う、という形が多いので自動生成しやすいです。

## 9. テクスチャを増やす方法

画像は大きく 3 パターンあります。

### 1. 完全に手描きする

見た目差分が大きいなら、この方法が向いています。

例:

- Tier ごとに模様が違う
- 発光部分の形が違う
- レア度ごとに装飾が増える

### 2. ベース画像から色違いを作る

今回のような「同じ形で色だけ違う」に一番向いています。

例:

- 赤・青・緑の結晶
- copper / iron / gold / diamond で色だけ変える素材

この場合は、ベース画像を 1 枚作って、外部スクリプトで色変換する方法が楽です。

### 3. 一部だけ手直しする半自動

ベース色はスクリプトで変えて、最後に数枚だけ手で調整する方法です。

大量にあるときは、このやり方がかなり現実的です。

## 10. テクスチャ自動化でよく使う流れ

色差分の PNG を増やすなら、だいたい次の流れが分かりやすいです。

1. 元になるテンプレ画像を 1 枚作る
2. 変えたい色の一覧を決める
3. Python などで画像を読み込む
4. 指定色だけを置き換える
5. ファイル名規則に従って書き出す

この方式なら、たとえば次のような生成ができます。

- `template_crystal.png` から `copper_crystal.png`
- `template_crystal.png` から `iron_crystal.png`
- `template_crystal.png` から `gold_crystal.png`

## 11. datagen と画像スクリプトをどうつなぐか

考え方としては、**同じ定義一覧を両方で参照する**のが理想です。

ただし最初は、そこまで厳密に共通化しなくても構いません。

実際には次のどちらかで十分です。

1. Java 側の Tier 一覧と、画像スクリプト側の色一覧を手でそろえる
2. 共通の JSON や CSV を置いて、両方がそれを読む

初心者のうちは、まず 1 の方が追いやすいです。

## 12. 少ない場合は無理に自動化しない

もし増やす数が 3 個や 4 個程度なら、全部を自動化する必要はありません。

その場合は次の方が早いです。

1. Java 登録は手で書く
2. JSON も手で書く
3. テクスチャだけ手で複製して色を変える

datagen やスクリプトは、**数が増えてから**入れても遅くありません。

## 13. 自動化を入れる目安

次のどれかに当てはまるなら、自動化を考える価値があります。

- 同じ種類を 5 個以上作る
- 今後さらに増える予定がある
- 名前規則がはっきりしている
- 手作業ミスが起きやすい

特に Tier が 4 段階、色が 6 種類、さらにブロック版もある、のような場合は、手でやるとかなり大変です。

## 14. 実践的なおすすめ順

このプロジェクトで始めるなら、次の順が現実的です。

1. まず 1 系列だけ手作業で 1 個作る
2. 名前規則と構成を固める
3. Java 登録を一覧ベースに寄せる
4. item model / block model / lang / recipe を datagen 化する
5. 最後に PNG の色差分スクリプトを入れる

最初から全部まとめて自動化するより、1 個手で作ってから規則を抽出する方が失敗しにくいです。

## 15. よくある失敗

### 名前規則が先に決まっていない

途中で命名を変えると、登録、モデル、lang、レシピ、テクスチャを全部直すことになります。

### PNG まで datagen だけで何とかしようとする

JSON と画像は役割が違うので、無理に 1 つへまとめない方が楽です。

### 先に大量生成してから設計を直す

最初に 1 個だけ手で通してから、自動化へ移る方が安全です。

### Java 登録だけ手作業のまま増やしすぎる

JSON を自動化しても、登録側がバラバラだと後から見返しにくくなります。

## 16. このプロジェクトでの実践的な結論

色違い・Tier違いのアイテムやブロックをいくつか増やすなら、次の考え方がおすすめです。

1. 登録と JSON は datagen を前提に設計する
2. テクスチャ PNG は別スクリプトか手作業で作る
3. まず 1 個作ってから規則化する

つまり、**「Mod 側で登録と JSON をまとめる」「画像だけ別に扱う」** のが、いちばん壊れにくく、後から増やしやすい構成です。
