# ItemStack 基本ガイド

この文書では、Minecraft Mod 開発でとてもよく出てくる `ItemStack` について、
初心者向けにまとめます。

`ItemStack` は、単に「アイテムの種類」を表すものではありません。

`ItemStack` は、次のような情報をまとめて持つ 1 つのまとまりです。

1. 何のアイテムか
2. 何個あるか
3. 追加データを持っているか

たとえば、
「丸石そのもの」と「丸石 32 個」は同じではありません。

Mod 開発では、多くの場合「Item」ではなく `ItemStack` を扱います。

## 1. Item と ItemStack の違い

まず最初に、ここを分けて理解するのが大事です。

### Item

- アイテムの種類そのもの
- 設計図に近い

### ItemStack

- 実際に今そこにあるアイテムの束
- 個数や追加データを持つ

たとえば、`Items.DIAMOND` はアイテムの種類です。

一方で、`new ItemStack(Items.DIAMOND, 3)` は
「ダイヤモンド 3 個の束」です。

## 2. なぜ ItemStack を使うのか

Minecraft のインベントリ、ドロップ、GUI、レシピ結果、BlockEntity 内の保存では、
実際には「何個あるか」が必要です。

そのため、次のような場面では `ItemStack` が使われます。

1. プレイヤーの手持ち
2. 機械のスロットの中身
3. ブロックが落とすアイテム
4. クラフト結果
5. ボタンや処理で消費する材料

## 3. 空の ItemStack という考え方

Minecraft では、スロットが空のときに `null` を使うのではなく、
**空の `ItemStack`** を使うことが多いです。

この考え方はとても重要です。

つまり、

- 何も入っていない
- `null` ではない
- でも中身は空

という状態があります。

そのため、中身確認では `null` 判定よりも
`isEmpty()` を意識する方が基本です。

## 4. よく使う基本操作

ItemStack では、まず次の操作を覚えるとかなり進めやすくなります。

1. 中身が空か調べる
2. 個数を調べる
3. 個数を増減する
4. コピーを作る
5. アイテム種類を調べる

### 空かどうか

```java
if (stack.isEmpty()) {
    // スロットが空のときの処理
}
```

### 個数を見る

```java
int count = stack.getCount();
```

### 個数を変える

```java
stack.setCount(16);
```

### コピーを作る

```java
ItemStack copiedStack = stack.copy();
```

### 種類を調べる

```java
if (stack.is(Items.COBBLESTONE)) {
    // 丸石のときの処理
}
```

## 5. `copy()` が大事な理由

初心者がつまずきやすい点として、同じ `ItemStack` をそのまま別の場所へ渡してしまうことがあります。

すると、意図せず同じ中身を共有してしまうことがあります。

たとえば、

1. 元のスタックを保持したい
2. 結果用に別のスタックを作りたい
3. 比較用に退避したい

というときは、`copy()` を使う方が安全です。

## 6. 個数の減少と `shrink()`

材料消費では、個数を 1 減らす処理がよく出ます。

そのときは `setCount(...)` を直接使うより、
意味が分かりやすいメソッドを使う方が追いやすいことがあります。

```java
stack.shrink(1);
```

これは「1 個減らす」という意味です。

逆に増やすときは `grow(...)` が使われます。

```java
stack.grow(2);
```

これは「2 個増やす」という意味です。

## 7. 0 個以下になったらどう考えるか

ItemStack を減らす処理では、
「0 個になったあとにどう扱うか」を意識する必要があります。

Minecraft 側の処理に任せられる場面もありますが、
独自ロジックでは次を確認した方が安全です。

1. 減らしたあと空になっていないか
2. スロット更新が必要か
3. BlockEntity の保存対象が変わったか

つまり、個数だけではなく、
**空になったこと自体を状態変化として見る** のが大事です。

## 8. 種類比較は `==` より `is(...)` を意識する

初心者のうちは、
「このスタックはこのアイテムか」をどう比較するか迷いやすいです。

ItemStack に対しては、まず `is(...)` を覚えると分かりやすいです。

```java
if (stack.is(Items.COAL)) {
    // 石炭のとき
}
```

これは「この ItemStack がそのアイテムか」を読みやすく書けます。

## 9. 同じアイテムでも完全に同じとは限らない

ここも重要です。

たとえば次の 2 つは、同じ見た目でも区別が必要になることがあります。

1. 普通のアイテム
2. CustomData を持ったアイテム

つまり、「アイテムの種類が同じ」だけでは不十分な場面があります。

特に独自データを持たせたアイテムでは、
種類だけでなく追加データも見て判断する必要があります。

## 10. ItemStack に追加データを持たせる場面

1. 強化段階を持たせたい
2. 所有者情報を持たせたい
3. 独自の進行状態を持たせたい
4. 特別な見た目や動作条件を持たせたい

Minecraft 1.21.1 では、従来の NBT という考え方に加えて、
`CustomData` を意識する場面があります。

このあたりの保存方式そのものは、詳しくは
[docs/nbt-guide.md](docs/nbt-guide.md)
と合わせて見ると整理しやすいです。

## 11. ItemStack はどこでよく出てくるか

### プレイヤーが手に持っている物

右クリック処理や使用処理では、手に持っている物が `ItemStack` で渡されます。

### GUI のスロットの中身

Menu や BlockEntity が持つ各スロットの中身は、基本的に `ItemStack` です。

### BlockEntity の保存対象

機械や保管ブロックの在庫は、最終的に `ItemStack` 群として考えることになります。

### レシピ結果

クラフトや独自加工の結果も `ItemStack` です。

## 12. GUI と ItemStack の関係

インベントリ付き GUI では、画面に見えている 1 マス 1 マスの中身は
それぞれ `ItemStack` です。

ここでの役割分担は次の通りです。

1. BlockEntity が本当の `ItemStack` を持つ
2. Menu がそれをスロットとして扱う
3. Screen が見た目を表示する

つまり、GUI を作るときに本当に動いている中心データは `ItemStack` です。

インベントリ付き GUI 全体の話は
[docs/inventory-gui-guide.md](docs/inventory-gui-guide.md)
と一緒に読むとつながりやすいです。

## 13. BlockEntity 内で考えるときの視点

BlockEntity にインベントリを持たせる場合は、
「中に複数の ItemStack が入っている」と考えます。

たとえば 3 スロット機械なら、概念的には次のように整理できます。

1. 0 番スロットの ItemStack
2. 1 番スロットの ItemStack
3. 2 番スロットの ItemStack

つまり、機械の在庫は
「複数の ItemStack の集まり」です。

## 14. ItemStack を直接いじるときの注意

ItemStack は状態を持っているので、
直接変更する処理では注意が必要です。

### 注意 1

元のスタックを壊したくないなら `copy()` を使う

### 注意 2

個数変更後に更新通知や保存が必要か考える

### 注意 3

空スタックの扱いを忘れない

### 注意 4

独自データが付いているなら種類だけで判定しない

## 15. よくある処理パターン

### 1 個だけ消費する

```java
if (!stack.isEmpty()) {
    stack.shrink(1);
}
```

### 一定個数あるか確認する

```java
if (!stack.isEmpty() && stack.getCount() >= 8) {
    // 8 個以上必要な処理
}
```

### 空なら新しく入れる

```java
if (stack.isEmpty()) {
    stack = new ItemStack(Items.COBBLESTONE, 1);
}
```

### 同じ種類なら個数を増やす

```java
if (stack.is(Items.COBBLESTONE)) {
    stack.grow(1);
}
```

## 16. `new ItemStack(...)` を使う場面

新しいアイテム束を作るときは `new ItemStack(...)` を使います。

たとえば次のような場面です。

1. レシピ結果を返す
2. GUI の出力スロットへ結果を入れる
3. ドロップ用アイテムを作る
4. 初期状態の中身を入れる

```java
ItemStack resultStack = new ItemStack(Items.IRON_INGOT, 4);
```

## 17. 最大スタック数も意識する

全てのアイテムが 64 個まで積めるとは限りません。

そのため、独自の詰め込み処理や移動処理では、
「まだこのスタックに積める余地があるか」を意識する必要があります。

特に GUI の移動処理や独自機械の自動搬入では重要です。

## 18. ItemStack と保存処理

ItemStack を扱うコードでは、
「見た目だけ変えた」のか「本当の中身を変えた」のかを意識する必要があります。

BlockEntity の中の ItemStack を変えたなら、
保存対象が変わったと考えるべきです。

つまり、機械の中身を変更した処理では、
そのあとに状態更新や保存反映が必要になることがあります。

## 19. ItemStack とクライアント / サーバー

ここも大事です。

見た目として表示される ItemStack はクライアントでも見えますが、
本当の中身の確定はサーバー側で考える方が安全です。

たとえば、

1. ボタンを押した
2. 材料を 1 個消費する
3. 結果を出力スロットへ入れる

このような処理は、見た目だけで終わらせず、
サーバー側で本当の ItemStack を更新する必要があります。

クライアントとサーバーの考え方は
[docs/client-server-basics.md](docs/client-server-basics.md)
も合わせて見ると整理しやすいです。

## 20. よくあるミス

### `null` で空判定しようとする

まずは `isEmpty()` を疑う方が Minecraft では自然です。

### 元の ItemStack をそのまま使い回してしまう

必要に応じて `copy()` を使います。

### 個数を減らしたあとに空になったことを見落とす

空スタック化は大きな状態変化です。

### 種類だけ見て、追加データの違いを無視する

独自アイテムでは誤判定の原因になります。

### クライアント側だけで在庫変更を完結させようとする

本当の中身はサーバー側で更新する必要があります。

## 21. 最初に意識するとよいこと

ItemStack を使うときは、まず次の 4 つを順番に考えると整理しやすいです。

1. これは空の可能性があるか
2. 個数を変える必要があるか
3. コピーが必要か
4. 種類だけ見れば足りるか

この 4 つを毎回確認するだけでも、かなり事故が減ります。
