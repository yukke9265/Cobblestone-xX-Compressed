# インベントリ付き GUI 手順

この文書では、このプロジェクトで
「アイテムを入れられる機械 GUI」を作るときの流れをまとめます。

ここでいうインベントリ付き GUI とは、たとえば次のようなものです。

- 入力スロットがある機械
- 出力スロットがある機械
- 燃料スロットがある炉風ブロック
- 中にアイテムを保存できる保管ブロック

ボタンだけの GUI と違って、インベントリ付き GUI では
**アイテムの中身そのもの** を扱います。

そのため、見た目だけではなく、BlockEntity、Menu、保存処理、スロット構成を
一緒に考える必要があります。

## 1. まず役割を分ける

インベントリ付き GUI では、次の 4 つを分けて考えると整理しやすいです。

### Block

- 右クリックの入口
- GUI を開くきっかけ

### BlockEntity

- 機械の中に入っているアイテム
- 燃焼時間や進行度
- 保存 / 読み込み

### Menu

- スロットの配置
- プレイヤーインベントリとの接続
- Shift クリック移動の処理

### Screen

- 背景画像
- スロットの見え方
- バーやラベル
- ボタンなどの見た目

整理すると、次の理解が基本です。

- BlockEntity: 本当の中身
- Menu: スロットの橋渡し
- Screen: 見た目

## 2. ボタン GUI と何が違うのか

ボタンだけの GUI は、極端に言えば「押した」という操作だけを扱えば作れます。

一方で、インベントリ付き GUI では次を扱います。

1. 何個アイテムが入っているか
2. どのスロットに何が入るか
3. プレイヤーインベントリとの移動
4. ワールドを閉じても中身が残るか

そのため、インベントリ付き GUI は
**BlockEntity に中身を持たせるところから始める** のが基本です。

## 3. 何を BlockEntity に持たせるか

最初に決めるのは、その機械が何スロット持つかです。

たとえば独自かまど風なら、次の 3 つが分かりやすいです。

1. 入力スロット
2. 燃料スロット
3. 出力スロット

保管箱風なら、次のようにもできます。

1. スロット 1 個だけ
2. スロット 9 個
3. スロット 27 個

初心者向けには、まずは **1 スロットか 3 スロット** が追いやすいです。

## 4. 先に決めておくべきこと

インベントリ GUI を作る前に、次を決めておくと作業が楽です。

1. 機械スロット数
2. 各スロットの役割
3. どのアイテムを入れられるか
4. 出力スロットから手で入れられるか
5. GUI 上の表示位置

これを決めないまま進むと、Menu 側と Screen 側で座標や意味がずれやすいです。

## 6. BlockEntity 側で必要になるもの

インベントリ付き GUI では、BlockEntity 側に少なくとも次が必要です。

1. アイテムを保存する場所
2. 保存 / 読み込み
3. GUI を開くための MenuProvider

### BlockEntity 側の仕事

- アイテムを本当に保持する
- ワールド保存時に中身も一緒に保存する
- GUI を開いたときに Menu を作れるようにする

ここで大事なのは、**Menu が本体の在庫を持つのではない** という点です。

本当の中身は BlockEntity 側です。

## 7. インベントリの中身をどう持つか

初心者向けには、まず「ItemStack の配列やリストを持つ」と考えると分かりやすいです。

実装の仕方はいくつかありますが、最初に理解したいのは考え方です。

たとえば 3 スロット機械なら、概念的にはこうです。

1. 0 番: 入力
2. 1 番: 燃料
3. 2 番: 出力

つまり、スロット番号と役割の対応を先に固定しておきます。

## 8. 保存と読み込みが必要

インベントリ付き GUI では、ワールドを閉じても中身が消えないようにする必要があります。

そのため、BlockEntity の `saveAdditional(...)` と `loadAdditional(...)` が重要です。

ここで保存するのはたとえば次のようなものです。

1. 各スロットの ItemStack
2. 燃焼時間
3. 進行度
4. 稼働状態

インベントリを持ち始めると、単なる整数より保存対象が増えるので、
「GUI を作る」より先に「中身を保存できる」ことを確認した方が安全です。

## 9. Menu 側でやること

Menu は、インベントリ付き GUI で特に重要です。

主な役割は次の通りです。

1. 機械スロットを追加する
2. プレイヤーインベントリを追加する
3. `stillValid(...)` を書く
4. `quickMoveStack(...)` を書く
5. 必要なら `ContainerData` を追加する

### Menu を一言で言うと

Menu は、**どこにどんなスロットがあるかを決めるクラス** です。

Screen がスロットを作るのではなく、Menu がスロットを持ちます。

## 10. スロット追加の考え方

インベントリ GUI では、Menu のコンストラクタで `addSlot(...)` を呼んでいきます。

最初に追加するのは、普通は機械スロットです。

そのあとでプレイヤーインベントリを追加します。

流れは次のようにすると分かりやすいです。

1. 機械スロットを上側へ置く
2. プレイヤーの 3 段インベントリを下へ置く
3. ホットバーを一番下へ置く

## 11. 座標の考え方

バニラ系 GUI の座標は、だいたい次のイメージで考えると分かりやすいです。

- スロット 1 個は 18 x 18 間隔
- 機械スロットは中央寄り
- プレイヤーインベントリは下側

たとえば 3 列の機械スロットなら、横方向に 18 ずつずらして並べます。

プレイヤーインベントリも同じ考え方で並べます。

## 12. プレイヤーインベントリを追加する理由

機械の中身だけ見えても、プレイヤーの手持ちが見えないと実際には使いにくいです。

そのため、多くのインベントリ GUI では、次をまとめて表示します。

1. 機械側スロット
2. プレイヤーインベントリ 27 マス
3. ホットバー 9 マス

つまり、見た目は 1 枚の GUI でも、
中では「機械」と「プレイヤー」を 1 つの Menu でつないでいます。

## 13. `stillValid(...)` は何を確認するか

インベントリ付き Menu では、プレイヤーがその GUI を開いたままでよいかを確認する必要があります。

たとえば次のような観点です。

1. 対象 BlockEntity がまだ存在するか
2. プレイヤーが遠すぎないか
3. ブロックが別のものに変わっていないか

最初は簡易的に true でも作れますが、
実用的には「そのブロックの近くにいる間だけ有効」にする方が自然です。

## 14. Shift クリック移動は後から必要になる

通常クリックだけなら GUI は動きますが、
実際に使いやすくするには `quickMoveStack(...)` が必要です。

これは Shift クリック時に、アイテムを自動で別領域へ送るための処理です。

たとえば次のような動きです。

1. 機械スロットから Shift クリックしたらプレイヤー側へ送る
2. プレイヤー側から Shift クリックしたら、入れられる機械スロットへ送る
3. 入れられないなら何もしない

## 15. `quickMoveStack(...)` を書く前に決めること

このメソッドを書く前に、先に次を決めておく必要があります。

1. 機械スロットの範囲
2. プレイヤーインベントリの範囲
3. ホットバーの範囲
4. 入力可能なスロット
5. 出力専用スロット

つまり、**どの index がどの領域なのか** をはっきり決めておくことが先です。

## 16. 典型的なスロット範囲の考え方

たとえば 3 スロット機械なら、次のような整理がよく使われます。

1. 0 から 2: 機械スロット
2. 3 から 29: プレイヤーインベントリ
3. 30 から 38: ホットバー

このように番号を固定しておくと、`quickMoveStack(...)` がかなり書きやすくなります。

## 17. 出力スロットは入力と扱いが違う

インベントリ GUI で初心者が混乱しやすい点の 1 つが、出力スロットです。

出力スロットは普通、次のような扱いにします。

1. プレイヤーが手で入れられない
2. 処理結果だけが入る
3. Shift クリックで持ち出せる

つまり、全スロットを同じ扱いにしない方が自然です。

## 18. Screen 側では何をするか

Screen 側では、インベントリそのものを定義しません。

Screen 側でやることは、主に次です。

1. 背景画像を描く
2. タイトルを描く
3. バーやラベルを描く
4. ボタンを置く
5. ツールチップを出す

つまり、スロット枠の見た目は背景画像に描かれていても、
本当のスロット定義は Menu 側です。

## 19. スロットの見た目はどう決まるか

スロットの当たり判定や中身は Menu が持っています。

一方で、どこにスロット枠の絵が見えるかは Screen の背景画像で決まります。

だから、次の 2 つがずれると見た目が壊れます。

1. Menu の `addSlot(...)` で置いた座標
2. 背景画像のスロット枠の位置

インベントリ GUI では、この座標合わせがとても大事です。

## 20. ボタンとインベントリを一緒に置くときの考え方

すでにボタンを実装できているなら、次はボタンとインベントリを同じ Screen に置くことになります。

このときの整理は次の通りです。

### ボタン

- Screen の `init()` で置く
- クライアント側の見た目部品

### インベントリスロット

- Menu のコンストラクタで追加する
- サーバー側の中身とつながる

つまり、同じ GUI に見えても、ボタンとスロットは役割が違います。

## 21. BlockEntity と Menu の結び付き

インベントリ付き GUI では、BlockEntity が持つ本当の中身を Menu が扱える必要があります。

そのため、BlockEntity 側で `createMenu(...)` を実装して、対応する Menu を返します。

この流れは次の通りです。

1. Block が右クリックを受ける
2. BlockEntity を取得する
3. BlockEntity が `createMenu(...)` で Menu を返す
4. Menu がその BlockEntity の中身を使ってスロットを構成する

## 22. Menu と Screen の結び付き

Menu と Screen の結び付きは、クライアント側の `RegisterMenuScreensEvent` で行います。

つまり、インベントリ付き GUI でも基本は同じです。

1. サーバー側で Menu を開く
2. クライアント側で、その MenuType に対応する Screen を出す

ボタン GUI からここは変わりません。

## 23. 値同期はどう考えるか

インベントリ GUI では、アイテム中身だけでなく、進行度や燃焼時間もよく表示します。

そのときは `ContainerData` が便利です。

役割は次のように考えると分かりやすいです。

1. BlockEntity が本当の値を持つ
2. Menu が同期用に受け取る
3. Screen が描画に使う

つまり、Screen が直接 BlockEntity を操作しにいくのではなく、
Menu を通して見る形に寄せると整理しやすいです。

## 24. 実装順はどうするか

初心者向けには、次の順で進めるのが安全です。

1. BlockEntity にスロット構成を決める
2. 保存 / 読み込みを実装する
3. Menu に機械スロットを追加する
4. プレイヤーインベントリとホットバーを追加する
5. Screen に背景画像を置く
6. スロット枠の見た目と座標を合わせる
7. `quickMoveStack(...)` を実装する
8. 最後にボタンや進行バーを足す

この順にすると、どこで壊れたか切り分けやすいです。

## 25. 最初の題材としておすすめの GUI

最初から独自かまどを完成版にしなくても構いません。

まずは次のような段階が追いやすいです。

### 段階 1

- スロット 1 個だけの保管 GUI

### 段階 2

- 入力 1 個、出力 1 個の簡易処理 GUI

### 段階 3

- 入力、燃料、出力の 3 スロット GUI

この順で進めると、インベントリ関連の考え方がかなり固まります。

## 26. よくあるミス

### Screen 側でスロットを作ろうとする

スロット定義は Menu 側です。

### BlockEntity に中身を持たせず Menu だけで完結させようとする

本当の在庫は BlockEntity 側へ持たせる方が自然です。

### 保存 / 読み込みを忘れる

ワールドを開き直したときに中身が消えます。

### 機械スロットとプレイヤーインベントリの index 管理が曖昧

`quickMoveStack(...)` が壊れやすくなります。

### スロット座標と背景画像の枠がずれている

見た目とクリック位置が食い違います。

### 出力スロットまで普通の入力スロットと同じ扱いにする

意図しないアイテム投入が起きやすいです。

## 27. 確認するときの順番

インベントリ付き GUI を作ったら、次の順で確認すると切り分けしやすいです。

1. GUI が開くか
2. プレイヤーインベントリが見えるか
3. 機械スロットが見えるか
4. 通常クリックでアイテムを置けるか
5. ワールドを開き直しても中身が残るか
6. Shift クリックで移動できるか
7. 入れてはいけないスロットへ入らないか
8. 進行バーや燃焼バーがあれば値が反映されるか

## 28. このプロジェクトで次にやるとよいこと

今の炉 GUI をインベントリ付きへ進めるなら、次の順が自然です。

1. `CobblestoneFurnaceBlockEntity` に機械スロットの中身を持たせる
2. `CobblestoneFurnaceMenu` に機械スロットとプレイヤーインベントリを追加する
3. `CobblestoneFurnaceScreen` の背景画像にスロット枠を用意する
4. `quickMoveStack(...)` を実装する
5. 最後に燃焼バーや進行バーを足す

ここまでできると、「ボタンがある GUI」から「本当に機械として使える GUI」へ進めます。
